# Архитектура и компромиссы

В данном документе описаны архитектурные решения, обоснование выбора стека, реализованные механизмы безопасности и компромиссы, допущенные в рамках тестового задания.

## Выбор стека

- **Vite + React 19 + TypeScript** — оптимальный выбор для быстрого создания производительных SPA. TypeScript обязателен при работе со строгими типами библиотек TON (`Address`, `Cell`, `SendMode`, `KeyPair`): компилятор ловит ошибки типов ещё до запуска.
- **Vanilla CSS с переменными** — намеренный отказ от UI-фреймворков (Tailwind, MUI). Результат: ~330 строк CSS, полный контроль над темой, ноль лишних зависимостей.
- **Официальные TON-библиотеки** (`@ton/ton`, `@ton/core`, `@ton/crypto`) — поддерживаются TON Foundation, являются индустриальным стандартом для новых DApp. Старый `tonweb` не рассматривался.
- **Orbs Network TON Access** (`@orbs-network/ton-access`) — предоставляет децентрализованный доступ к публичным API-нодам Testnet/Mainnet без регистрации API-ключей. Используется в `services/ton.ts` через `getHttpEndpoint({ network: 'testnet' })` для динамического получения эндпоинта. Альтернативы (TonCenter, TonAPI) имеют жёсткие rate limits на бесплатных тарифах.
- **Стейт-машина в `App.tsx`** вместо `react-router` — оправдана малым числом экранов (`INIT → DASHBOARD ↔ SEND / RECEIVE`).

## Архитектура без бэкенда (Self-Custodial & No-Backend)

### Хранение ключей
Seed-фраза генерируется локально (`mnemonicNew` из `@ton/crypto`), шифруется XOR-шифром с паролем пользователя и сохраняется в `localStorage`.

**Компромисс:** XOR — слабый шифр, уязвимый к known-plaintext атаке. В production применяется **WebCrypto API**: PBKDF2 (≥100 000 итераций) для вывода ключа из пароля + AES-GCM для шифрования в IndexedDB. Для тестового задания реализован упрощённый механизм, демонстрирующий понимание концепции "кошелёк закрыт паролем".

### История транзакций
Данные берутся напрямую из блокчейна через `client.getTransactions()`. Транзакции парсятся в `useMemo` один раз — результат переиспользуется и для рендера, и для поиска (исключает дублирование логики).

**Компромисс:** Сырой RPC TON возвращает структуру `inMessage / outMessages` без агрегации. В полноценном кошельке используют индексаторы (**TonAPI**, **TONCenter v3**) — они отдают человекочитаемые события (`Transfer`, `Swap`, `NFT`). Для тестового задания поддержан только TON-перевод.

### Rate limiting
Все запросы к ноде проходят через очередь `enqueue()` с задержкой 1.1 с между вызовами. Это исключает HTTP 429 при пакетной загрузке баланса + истории на Dashboard.

## UX: Двухшаговая отправка

Отправка реализована в два шага: **форма → экран подтверждения**. На экране подтверждения пользователь видит:
- Полный адрес получателя (не обрезанный)
- Сумму
- **Оценку комиссии сети** (запрос к ноде)
- Повторное предупреждение при address spoofing

**Оценка комиссии** (`estimateTransferFee` в `services/ton.ts`):
1. Получает актуальный `seqno` с ноды.
2. Строит transfer-ячейку с dummy-ключом (64 нулевых байта).
3. Вызывает `client.estimateExternalMessageFee(..., { ignoreSignature: true })` — нода считает газ без проверки подписи.
4. Суммирует `in_fwd_fee + storage_fee + gas_fee + fwd_fee`.
5. При любой ошибке возвращает fallback `≈0.005 TON` с пометкой "приблизительно".

## Защита от Address Spoofing

**Вектор атаки:** злоумышленник генерирует адрес с тем же началом и концом, что и доверенный контакт жертвы, и отправляет микротранзакцию. Пользователь копирует "знакомый" адрес из истории — и отправляет средства мошеннику.

**Реализация в `src/services/security.ts`:**

При вводе адреса в форму отправки:
1. Адрес валидируется через `Address.parse()` из `@ton/core`.
2. Сравнивается с набором "известных" адресов из последних 50 транзакций (входящих и исходящих).
   - **100% совпадение** — адрес доверенный, предупреждений нет.
   - **Совпадают первые 5 и последние 5 символов, но середина отличается** — красный блок с визуальной подсветкой подозрительной части; кнопка отправки заменяется на "Всё равно отправить".
   - **Новый адрес** — мягкое информационное предупреждение.
3. Проверка повторяется на экране подтверждения.

**Почему 5 символов:** типичная атака использует совпадение первых 4–6 символов. Порог в 5 даёт разумный баланс между чувствительностью (не пропустить атаку) и специфичностью (не ложно срабатывать на случайные совпадения в коротких адресах).

## Компромисс: QR-код через внешний сервис

QR-код в экране Receive генерируется через `api.qrserver.com`. Адрес кошелька публичен по природе (пользователь намеренно им делится), поэтому утечки приватных данных нет. Тем не менее, для production self-custodial кошелька правильнее генерировать QR локально (например, `qrcode.react`) — это исключает зависимость от внешнего сервиса и любые запросы к третьим сторонам.

## Дальнейшие улучшения (Production-grade)

1. **WebCrypto API** — PBKDF2 (≥100 000 итераций) + AES-GCM для шифрования seed-фразы в IndexedDB.
2. **TonAPI / индексатор** — человекочитаемая история с поддержкой Jetton-токенов, NFT, комиссий.
3. **Wallet V5** — последний стандарт кошелька TON с gasless-транзакциями. В проекте выбран `WalletV4R2` как наиболее проверенный и совместимый.
4. **QR-код локально** — заменить `api.qrserver.com` на `qrcode.react` для полного no-backend.
5. **Интеграционные тесты** — отдельный CI-пайплайн с реальным testnet-кошельком.
6. **react-router** или Context API — актуально при росте числа экранов; текущая стейт-машина в `App.tsx` оправдана малым размером приложения.
7. **Оптимизация бандла** — итоговый размер ~716 KB (gzip: ~214 KB) обусловлен TON-библиотеками, которые включают WebAssembly-криптографию. В production решается через `React.lazy` + `Suspense` для экранов Send и Receive: пользователь загружает тяжёлую криптографическую часть только в момент первого открытия этих экранов.
